# คู่มือการใช้งานระบบ QSMS Rework Management System (ฉบับละเอียด)

ระบบนี้ถูกออกแบบมาเพื่อจัดการและติดตามกระบวนการทำงาน Rework สินค้า โดยเชื่อมต่อข้อมูลระหว่างหน้าเว็บ (React) และฐานข้อมูล (Google Sheets) อย่างไร้รอยต่อ

---

## 1. การเข้าสู่ระบบ (Login)
- **สิทธิ์การใช้งาน (Roles):**
    - **ADMIN / QSMS:** เข้าถึงได้ทุกเมนู รวมถึงแดชบอร์ดสถิติ และสามารถเข้าใช้งานโมดูล Roster ตารางเวรได้เต็มระบบ
    - **OPERATOR / WFG / PDB (สิทธิ์ปฏิบัติการ Rework):** ยุบรวมสิทธิ์เป็นกลุ่มผลิตและคลังสินค้า โดยจำกัดการใช้งาน **เฉพาะโมดูล Rework** เท่านั้น (ซ่อนโมดูล Roster ทั้งหมด)
        - สามารถกดเพิ่มงานใหม่ (Add Case) และค้นหาภาพรวม (Overall) ได้
        - สามารถใส่รายการรายละเอียดสินค้า ข้อมูลความเสียหาย และอัปโหลดภาพถ่ายประกอบได้
        - สามารถกดบันทึกและอัปเดตสถานะงานเป็น "กำลังดำเนินการ (In-Progress)" หรือ "รอประเมินราคา (Awaiting Valuation)"
        - **ข้อจำกัด:** ถูกซ่อนและล็อกปุ่มแก้ไขค่าใช้จ่ายราคาทั้งหมด (No Pricing/Cost Access) รวมถึงไม่มีสิทธิ์ทำการ Export ข้อมูลหรือเรียกดูแดชบอร์ด
    - **FINANCE (แผนกการเงิน):** จำกัดสิทธิ์เฉพาะการดูตารางภาพรวม (Overall) และทำการกรอกประเมินราคางาน (Rework Cost และ Labor Rate) ในช่องใบงาน เพื่อส่งมอบงานเป็นขั้นตอนเสร็จสมบูรณ์ (Completed) แต่จะไม่สามารถกดเพิ่มงาน (Add Case) หรือแก้ไขรายละเอียดสินค้าด้านในได้
- **วิธีการล็อกอิน:** กรอกรหัสผู้ใช้งานและรหัสผ่านจากระบบกลาง หรือใช้ **บัญชีสำหรับการทดสอบภายใน (Mock Accounts)** ดังนี้:
    - **บัญชี QSMS/Admin:** Username: `qsms`, Password: `Qsms123`
    - **บัญชี Operator/WFG/PDB:** Username: `operator`, Password: `Operator123`
    - **บัญชี Finance:** Username: `finance`, Password: `Finance123`

---

## 2. หน้าหลัก: ภาพรวมรายการ (Overall)
หน้านี้เปรียบเสมือนศูนย์กลางการติดตามงานทั้งหมด
- **การทักทาย:** ระบบจะทักทายตาม Role ที่ Login เข้ามา (เช่น "สวัสดี QSMS")
- **การค้นหาและกรอง:**
    - ค้นหาด้วยคำสำคัญ (Search)
    - กรองตามสถานะ (Pending, In-Progress, Awaiting Valuation, Completed)
    - กรองขั้นสูง (Advanced Filter) ตามแหล่งที่มา, สาเหตุ, ผู้รับผิดชอบ และช่วงวันที่
- **สรุปสถิติเบื้องต้น:** แสดงจำนวนงานในแต่ละสถานะที่ด้านบนของหน้า

---

## 3. การเพิ่มงานใหม่ (Add Case)
ใช้สำหรับลงบันทึกงาน Rework ที่เกิดขึ้นใหม่
- **ข้อมูลพื้นฐาน:** เลือกแหล่งที่มาของงาน (Source)
- **ข้อมูลสินค้า (Items):**
    - **Item Number:** กรอกหมายเลขสินค้า (รองรับตัวอักษรพิเศษ . - / )
    - **Auto-Fill:** ระบบจะดึงชื่อสินค้าให้อัตโนมัติหากเคยมีข้อมูลในฐานข้อมูล
    - **Batch No:** กรอกหมายเลขล็อต
    - **Reason & Responsible:** เลือกสาเหตุและหน่วยงานที่รับผิดชอบจากเมนูตัวเลือก
- **รูปภาพ:** สามารถอัปโหลดรูปภาพหลักฐานประกอบได้
- **การบันทึก:** กด "บันทึกรายการ" เพื่อส่งข้อมูลไปยัง Google Sheets

---

## 4. การจัดการและอัปเดตรายการ (Update & Manage)
เมื่อคลิกที่รายการในหน้า Overall จะเปิดหน้าต่างแก้ไข (Update Modal)
- **การอัปเดตสถานะ:** เปลี่ยนสถานะงานตามความคืบหน้า (รอดำเนินการ ➡️ กำลังทำ ➡️ รอประเมิน ➡️ เสร็จสิ้น)
- **การระบุวิธีแก้ไข:** ใส่รายละเอียดการแก้ไขในช่อง "Resolution Method"
- **การระบุค่าใช้จ่าย:** (สำหรับ Finance/Admin) ใส่ราคาที่ช่อง "Rework Cost"
- **การลบรายการ:** สามารถกด "ลบรายการ" หากข้อมูลผิดพลาด (ระบบจะมีระบบยืนยันก่อนลบ)

---

## 5. การออกรายงาน (Export Report)
ภายในหน้าต่าง Update Modal จะมีปุ่มสำหรับส่งออกข้อมูล
- **รูปแบบ:** รองรับทั้ง PDF และ PNG
- **จุดเด่น:** รายงานจะแสดงข้อมูลล่าสุดที่แก้ไขทันที รวมถึงรูปภาพประกอบที่อัปโหลดไว้ เพื่อใช้เป็นเอกสารอ้างอิงที่เป็นทางการ

---

## 6. แดชบอร์ดวิเคราะห์ผล (Dashboard)
(เฉพาะ Admin และ QSMS)
- **KPI Summary:** ดูภาพรวมประสิทธิภาพการทำงาน
- **Chart Analysis:** วิเคราะห์สาเหตุที่เกิด Rework บ่อยที่สุด และสัดส่วนความรับผิดชอบของแต่ละหน่วยงาน เพื่อนำไปสู่การปรับปรุงคุณภาพสินค้า

---

## 7. ระบบการทดสอบอัตโนมัติ (Automated Testing Pipeline)
สำหรับนักพัฒนาหรือผู้ดูแลระบบที่ต้องการตรวจสอบความเสถียรของแอปพลิเคชันก่อนนำไปใช้งานจริง:
- **Unit & Integration Tests (Vitest):** ใช้ตรวจสอบการทำงานของ Helpers, ฟังก์ชันการจัดเรียงสถานะ และฟังก์ชันการตรวจสอบความถูกต้องของฟอร์ม (Validation) 
  * คำสั่งรัน: `npm run test` (รันการทดสอบทั้งหมด 51 รายการ)
- **End-to-End (E2E) Tests (Playwright):** จำลองการกดเบราว์เซอร์จริง เช่น ล็อกอิน, กรอกฟอร์ม Rework, เลือกโมดอล และตรวจเช็คผลการบันทึกข้อมูล
  * คำสั่งรัน: `npm run test:e2e`

## 🛠️ รายละเอียดการอัปเดตล่าสุด (Commit Summary)
ผมได้ทำการอัปเดตโค้ดขึ้น GitHub เรียบร้อยแล้ว โดยมีการปรับปรุงเพิ่มเติมดังนี้:
1.  **การยุบรวมบทบาทผู้ใช้งาน (Role Consolidation):** ปรับปรุงโครงสร้างสิทธิ์การเข้าใช้งานระหว่าง Operator, WFG และ PDB ให้มีระดับเดียวกัน โดยตัดการใช้งานใน Roster และจำกัดหน้าจอใน Rework ให้ทำการลงบันทึกและจัดการทรัพยากรได้โดยปราศจากการกรอกตัวเลขเงิน/ราคาใดๆ (No Pricing Access)
2.  **ระบบบัญชีทดสอบภายใน (Internal Mock Accounts):** สร้างระบบ Interceptor ดักจับ API การล็อกอินของ Next.js เซิร์ฟเวอร์ และทำการเซ็นโทเค็นความปลอดภัย (Crypto Sign Validation) อัตโนมัติด้วย `AUTH_TOKEN_SECRET` ทำให้ผู้ตรวจสอบสามารถใช้ Username/Password จำลอง (`qsms`, `operator`, `finance`) เพื่อทดสอบพฤติกรรมแต่ละ Role บนบราวเซอร์ได้ทันที
3.  **การจัดเรียงรายการเคส (Case Sorting Improvement):** ปรับแก้ให้เคสที่มีสถานะ `Pending` (รอดำเนินการ) ขึ้นแสดงบนสุดของตาราง Overall เสมอ ตามด้วย `In-Progress`, `Awaiting Valuation` และ `Completed` เพื่อไม่ให้งานค้างถูกดันตกไปด้านล่าง และเรียงตามลำดับวันที่ล่าสุดลงไป
4.  **ระบบการทดสอบอัตโนมัติ (Testing Pipeline):** จัดทำโครงสร้างระบบทดสอบ Vitest และ Playwright ครอบคลุมทั้งส่วน Unit tests (51 เคส) และ E2E tests (3 ชุดหลัก) เพื่อเพิ่มความมั่นใจในการพัฒนาโค้ดและป้องกัน Regression
5.  **ความเสถียร:** แก้ไขระบบการลบรายการให้ทำงานได้สมบูรณ์ (ปิดหน้าต่างอัตโนมัติและแสดงสถานะกำลังลบ)
6.  **ความยืดหยุ่น:** ปรับปรุงให้ช่อง Item Number และ Batch No รองรับตัวอักษรพิเศษได้หลากหลายมากขึ้น
7.  **ความสวยงาม:** ปรับปรุง Export Template ใหม่ให้เป็นสไตล์ Enterprise พร้อมภาษาไทยที่สมบูรณ์
8.  **ระบบผู้ใช้:** ปรับปรุงคำทักทายให้เปลี่ยนตาม Role ของผู้ใช้งานจริง
9.  **Build Fix:** แก้ไขปัญหาทางเทคนิคเพื่อให้ระบบทำงานได้รวดเร็วและไม่มีข้อผิดพลาดแฝง

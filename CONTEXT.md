# QSMS Domain Glossary

This document defines the canonical terminology used across the QSMS Rework Management System project.

### Item Master
A central reference directory of all items/products in the system. Each item is defined by three core properties:
- **Item Number**: A unique alphanumeric identifier for the item (e.g. `TEST-001`).
- **Item Code**: A unique numeric barcode identifier for the item (e.g. `123456789`).
- **Item Name**: The descriptive name of the item.

### Rework Item Granular Fields
Additional batch-specific information collected per item inside a Rework case, including:
- **Batch No. (หมายเลขล็อต)**: The production batch date.
- **Gallon Date (วันที่ผลิตแกลลอน)**: The date the plastic gallon was manufactured.
- **Box Number (เลขกล่อง)**: The box identifier containing the item.
- **Mold (แม่พิมพ์)**: The mold number used for production.
- **Line (สายการผลิต)**: The production line identifier.

### Incomplete Item
A record in the [Item Master](#item-master) that is missing one or more of the three core properties (having `null` or empty values).

### Complete Item
A record in the [Item Master](#item-master) that has all three core properties successfully populated.

### Strict Identity Conflict
A data integrity violation where a provided **Item Number** matches one record in the [Item Master](#item-master) while the provided **Item Code** matches a different record. This state is treated as a hard block to prevent identity fragmentation.

### Two-Way Autofill
A user interface interaction that automatically queries the [Item Master](#item-master) and fills in the remaining fields in the case form (with a 600ms debounce) when either the **Item Number** or **Item Code** is provided.

### Smart Master Upsert
The background synchronization logic that inserts new items into the [Item Master](#item-master) or updates existing [Incomplete Items](#incomplete-item) to populate missing fields, preventing duplicate records and ensuring data completeness.

### Unregistered Item
An item provided by the user that does not currently exist in the [Item Master](#item-master). These items are flagged for automatic registration upon successful case submission.

### Verification Lifecycle
The sequence of states an item entry goes through during validation:
- **Idle**: Waiting for user input.
- **Checking**: Verification in progress against the [Item Master](#item-master).
- **Verified**: Confirmed match found in the [Item Master](#item-master).
- **New**: Identified as an [Unregistered Item](#unregistered-item).
- **Conflict**: Encountered a [Strict Identity Conflict](#strict-identity-conflict).

### RT/RW
Common shorthand referring to the **Rework Ticket** (RT) or **Rework Document / Case** (RW). RT is used as a Case ID prefix for Customer cases, while RW is used for SFC (Shop Floor Control) cases.

### Case ID
A unique identifier for a rework case. It follows a **Hybrid Assignment** model:
- **Primary**: Assigned directly from the [RT/RW](#rtrw) document identifier composed of prefix (RW/RT), running number, and current year (e.g., `RW012-2026`).
- **Fallback**: Automatically generated by the system (e.g., `RWYYYYMMDD-XXX`) if no physical document ID is provided.

Once assigned, a **Case ID** is considered **Immutable** and unique; any attempt to submit a duplicate Case ID will be rejected by the system.

### Rework Item
A specific unit or batch within a [Case ID](#case-id) that requires rework. A single case may contain multiple items.

### Rework Reason & Subtype
A two-part classification of why an item requires rework. These are **Independent Fields**; while the Subtype provides more granular detail, it is not strictly constrained by the primary Reason in the system logic. Some reasons (such as 'รั่ว' or 'เปื้อน') trigger granular subtype selections.

### Cross-Item Link (Linked Source Item)
An optional relational reference for stained items ('เปื้อน'). If an item is stained, and another item in the same case is leaking ('รั่ว'), the user can check the link option and specify the leaking item as the root cause via the `linkedSourceId` field.

### Responsible Party & Subtype
A two-part classification of which entity or process is accountable for the rework. Like reasons, these are **Independent Fields** allowing for flexible reporting.

### Evidence Integrity
The requirement that every [Rework Item](#rework-item) must be accompanied by at least one visual record (image) during case creation. Images undergo client-side compression (target 300KB) to ensure efficient upload.

### PTT OR Rework Documents (OR Files)
An optional feature for cases where the customer name is "OR". Users can attach up to 2 OR files (.xlsx, .xls, .pdf, or .png) to accompany the rework case.

### Missing OR File Warning
A visual alert badge ("ขาดไฟล์ OR") displayed on the Case List Table for any case belonging to the customer "OR" that does not contain OR file attachments.

### Transaction Integrity
A submission policy ensuring that a [Case ID](#case-id) and its associated images are treated as a single atomic unit. If image uploads fail, the entire transaction is rolled back, and the Case is not recorded in the system, preventing incomplete records.

### DocAI RAG (Retrieval-Augmented Generation)
The internal AI module designed to read and ingest knowledge from unstructured documents (PDFs, Excel) into a Vector Database. Users can query this module to retrieve answers based strictly on the uploaded reference documents.

### Semantic Chunking
The process of breaking down large documents into contextually meaningful segments based on document structure (using `MarkdownTextSplitter` for Markdown headers) rather than fixed character counts. This prevents cutting off sentences mid-way and ensures that search queries match against complete, relevant topics before generating embeddings.

### Automated Metadata Extraction
A background process where the AI automatically analyzes an uploaded document to determine its category, relevant machine parts, or context. This metadata is stored alongside the chunks to act as filters, improving the accuracy of subsequent similarity searches without requiring manual user input.

### Direct Image Ingestion (Vision OCR)
The capability of the system to accept raw image files (JPG, PNG) as primary knowledge sources. The AI utilizes Vision models to perform Optical Character Recognition (OCR) and generate rich contextual descriptions of the visual content, which are then chunked and stored in the Vector Database just like text documents.

### RAG PDF Page-to-Image Ingestion
A preprocessing pipeline in RAG where PDF files are converted page-by-page into JPEG images on the client side using `pdfjs-dist`. These page images are uploaded to Supabase Storage (`rag_images`) and analyzed by Gemini's vision capability to produce rich markdown descriptions, preserving diagrammatic and visual details for embedding.

### RAG Bulk Document Deletion
A document management action (`bulk_delete_documents`) allowing users to check multiple documents in the checklist and delete them simultaneously. The database removes the metadata and all associated vector chunks via cascading foreign keys.

### Session Conversation Memory
A temporary context window maintained purely in the client-side UI (React State) during an active chat session. The frontend sends the recent history of messages (up to 5 messages) to the AI to allow for conversational follow-up questions, but this memory is intentionally discarded upon closing the interface to conserve token costs and minimize database complexity.

### Embedding
The process of converting a text chunk into a high-dimensional vector array.

### Jina AI Embeddings
The text embedding model (`jina-embeddings-v5-text-small`) used by the RAG module to generate 768-dimensional embeddings, which matches the dimensions of the Supabase `pgvector` database schema.

### Gemini Ingestion & Chat Models
The server-side Google Gemini models:
- **Ingestion (Parsing)**: Uses `gemini-2.5-flash` with vision capabilities to parse documents into Markdown.
- **Chat Interface**: Uses `gemini-3.1-flash-lite` to stream responses in real-time.

### Hybrid Search
A retrieval strategy that combines Vector Search (for semantic meaning) with Full-Text Search (for exact keyword matching). This ensures high accuracy when searching for highly specific technical terms, part numbers, or item codes that embedding models might otherwise misinterpret.

### Streaming Responses
A user experience pattern where the AI's response is sent to the client character-by-character in real-time, reducing perceived latency and mimicking natural typing behavior.

### Suggestion Chips
Context-aware, clickable buttons containing suggested follow-up questions generated by the AI based on the current conversation state, designed to guide users deeper into the topic without requiring manual typing. These are extracted from a hidden JSON block appended to the AI's response.

### RAG Feedback Loop
A continuous improvement mechanism where users can rate AI responses (Thumbs Up/Down). Feedback triggers an automated log entry in the `rag_feedback` table capturing the query, retrieved context, and the AI's response, allowing administrators to identify knowledge gaps or hallucination patterns.

### Nong Beepa Persona (AI Behavior)
The official persona for the QSMS DocAI. Nong Beepa is designed to be "Friendly, Helpful, & Empathetic." He communicates with a supportive, polite tone, frequently using polite particles (ครับ/ค่ะ) and emojis (🐝✨😊) to reduce user stress on the factory floor while delivering highly accurate technical assistance.

### Micro-Animations (Nong Beepa UI)
The standard for UI/UX interaction within the QSMS DocAI chat interface. The system leverages `framer-motion` (imported from `motion/react`) to provide smooth scrolling, bouncy typing indicators, and playful avatar hover effects. These animations are designed to make the AI feel alive, premium, and highly responsive.

### DocAI Deep Analysis
The analytical phase of QSMS DocAI that focuses on discovering patterns, summarizing common rework issues, and generating insights from ingested documents and operator feedback, rather than executing automated actions on the user's behalf. This approach optimizes token consumption by utilizing direct, single-turn LLM reasoning and structured search context instead of multi-turn agentic reasoning loops.

### DocAI Source Citation
The mechanism of attaching precise metadata references (such as document name, page number, section title, or original text snippet) directly to the generated AI response. This allows operators to instantly cross-reference and verify the validity of the technical rework instructions with the source files, eliminating guesswork and ensuring safety on the shop floor.

### React Hook Form & Zod Validation
The unified form validation layer implemented on the Rework module (Add Case tab). It replaces complex manual state tracking with schema-based verification, enforcing structural constraints (e.g. required fields, valid quantities, date formats) before submission.

### Zero-Width Space (\u200B) Word Wrapping
A formatting technique used in PDF/HTML layouts to resolve clipping of Thai text. Because Thai lacks spaces between words, rendering engines often fail to wrap lines correctly. Inserting zero-width spaces (`\u200B`) after characters provides the layout engine with artificial break points, preventing text truncation.

### Excel Export with Image Embedding
A data export format implemented via `exceljs` that compiles rework cases into a styled `.xlsx` file. It features structured headers (blue fill `#0066CC`, white bold text), wrapped text alignment, and automatically embeds up to 3 evidence images inside the "Images" column per item, adjusting the row height to 120px to display them cleanly.

### Reduced Motion Support
An accessibility design pattern implemented in `index.css` that detects OS-level reduced motion preferences using `@media (prefers-reduced-motion: reduce)` and forces transition and animation durations to 0.01ms, ensuring safety for users with vestibular disorders.

### Layout Shift Prevention
A layout optimization utilizing the CSS property `scrollbar-gutter: stable` to reserve space for scrollbars, preventing sudden page layout shifts when modals appear or disappear.
